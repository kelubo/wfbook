\chapter{Kubernetes 基础概念}

\section{什么是 Kubernetes}
\label{sec:what_is_k8s}

Kubernetes（简称K8s）是一个开源的容器编排平台，用于自动化部署、扩展和管理容器化应用。

Kubernetes 的主要功能：
\begin{itemize}
    \item \textbf{自动化部署}：自动部署应用容器
    \item \textbf{自动扩展}：根据负载自动增减容器数量
    \item \textbf{自愈能力}：自动重启失败的容器
    \item \textbf{服务发现}：自动发现和路由服务
    \item \textbf{负载均衡}：在多个容器间分配流量
\end{itemize}

\section{容器化基础}
\label{sec:containerization}

容器是一种轻量级的虚拟化技术，将应用及其依赖打包在一起。

容器与虚拟机的区别：
\begin{table}[htbp]
    \centering
    \caption{容器与虚拟机对比}
    \begin{tabular}{|c|p{5cm}|p{5cm}|}
        \hline
        \textbf{特性} & \textbf{容器} & \textbf{虚拟机} \\
        \hline
        隔离级别 & 进程级隔离 & 系统级隔离 \\
        \hline
        资源占用 & 低 & 高 \\
        \hline
        启动速度 & 秒级 & 分钟级 \\
        \hline
        镜像大小 & 小 & 大 \\
        \hline
        兼容性 & 需要兼容内核 & 完全独立 \\
        \hline
    \end{tabular}
\end{table}

\chapter{核心概念}
\label{chap:core_concepts}

\section{Pod}
\label{sec:pod}

Pod 是 Kubernetes 的最小部署单元，包含一个或多个容器。

Pod 的特点：
\begin{itemize}
    \item 多个容器共享网络和存储
    \item 容器间通过 localhost 通信
    \item Pod 是原子性的，一起创建和销毁
\end{itemize}

Pod 的类型：
\begin{itemize}
    \item \textbf{自主 Pod}：独立运行的 Pod
    \item \textbf{控制器管理的 Pod}：由控制器管理的 Pod
\end{itemize}

\section{ReplicaSet}
\label{sec:replicaset}

ReplicaSet 确保指定数量的 Pod 副本在运行。

ReplicaSet 的功能：
\begin{itemize}
    \item 维持指定数量的 Pod 副本
    \item 自动替换失败的 Pod
    \item 支持滚动更新
\end{itemize}

\section{Deployment}
\label{sec:deployment}

Deployment 是更高层的抽象，管理 ReplicaSet 和 Pod。

Deployment 的功能：
\begin{itemize}
    \item 声明式管理 Pod
    \item 支持滚动更新和回滚
    \item 支持暂停和继续部署
\end{itemize}

\section{Service}
\label{sec:service}

Service 为一组 Pod 提供稳定的访问入口。

Service 的类型：
\begin{table}[htbp]
    \centering
    \caption{Service 类型}
    \begin{tabular}{|c|p{8cm}|}
        \hline
        \textbf{类型} & \textbf{说明} \\
        \hline
        ClusterIP & 集群内部 IP，默认类型 \\
        \hline
        NodePort & 在节点端口上暴露服务 \\
        \hline
        LoadBalancer & 通过负载均衡器暴露服务 \\
        \hline
        ExternalName & 映射到外部服务 \\
        \hline
    \end{tabular}
\end{table}

\section{Namespace}
\label{sec:namespace}

Namespace 用于隔离集群资源。

Namespace 的用途：
\begin{itemize}
    \item 隔离不同团队或项目的资源
    \item 控制资源访问权限
    \item 资源配额管理
\end{itemize}

\section{kubeadm 安装}
\label{sec:kubeadm_install}

使用 kubeadm 安装 Kubernetes（CentOS Stream 9）：

\textbf{步骤1：准备环境}

禁用防火墙和 SELinux：
\begin{lstlisting}
systemctl stop firewalld && systemctl disable firewalld
setenforce 0
sed -i 's/^SELINUX=.*/SELINUX=disabled/' /etc/selinux/config
\end{lstlisting}

关闭 swap：
\begin{lstlisting}
swapoff -a
sed -i '/swap/d' /etc/fstab
\end{lstlisting}

设置内核参数：
\begin{lstlisting}
cat > /etc/sysctl.d/k8s.conf <<EOF
net.bridge.bridge-nf-call-ip6tables = 1
net.bridge.bridge-nf-call-iptables = 1
EOF
sysctl --system
\end{lstlisting}

\textbf{步骤2：安装 Docker}

\begin{lstlisting}
dnf install -y dnf-utils
dnf config-manager --add-repo https://download.docker.com/linux/centos/docker-ce.repo
dnf install -y docker-ce docker-ce-cli containerd.io
systemctl enable --now docker
\end{lstlisting}

\textbf{步骤3：安装 kubeadm、kubelet、kubectl}

\begin{lstlisting}
cat > /etc/yum.repos.d/kubernetes.repo <<EOF
[kubernetes]
name=Kubernetes
baseurl=https://packages.cloud.google.com/yum/repos/kubernetes-el7-x86_64
enabled=1
gpgcheck=1
repo_gpgcheck=1
gpgkey=https://packages.cloud.google.com/yum/doc/yum-key.gpg https://packages.cloud.google.com/yum/doc/rpm-package-key.gpg
EOF

dnf install -y kubelet kubeadm kubectl
systemctl enable --now kubelet
\end{lstlisting}

\textbf{步骤4：初始化集群}

\begin{lstlisting}
kubeadm init --pod-network-cidr=10.244.0.0/16
\end{lstlisting}

\textbf{步骤5：配置 kubectl}

\begin{lstlisting}
mkdir -p $HOME/.kube
cp -i /etc/kubernetes/admin.conf $HOME/.kube/config
chown $(id -u):$(id -g) $HOME/.kube/config
\end{lstlisting}

\textbf{步骤6：安装网络插件}

\begin{lstlisting}
kubectl apply -f https://raw.githubusercontent.com/flannel-io/flannel/master/Documentation/kube-flannel.yml
\end{lstlisting}

\section{kubeadm 安装（Ubuntu 26.04）}
\label{sec:kubeadm_install_ubuntu}

使用 kubeadm 安装 Kubernetes（Ubuntu 26.04）：

\textbf{步骤1：系统准备（所有节点）}

禁用 Swap：
\begin{lstlisting}
swapoff -a
sed -i '/\sswap\s/d' /etc/fstab
free -h   # 验证 Swap 应为 0
\end{lstlisting}

加载内核模块：
\begin{lstlisting}
cat <<EOF | sudo tee /etc/modules-load.d/k8s.conf
overlay
br_netfilter
EOF

modprobe overlay
modprobe br_netfilter
lsmod | grep -E "overlay|br_netfilter"   # 验证
\end{lstlisting}

配置内核网络参数：
\begin{lstlisting}
cat <<EOF | sudo tee /etc/sysctl.d/k8s.conf
net.bridge.bridge-nf-call-iptables  = 1
net.bridge.bridge-nf-call-ip6tables = 1
net.ipv4.ip_forward                 = 1
EOF

sysctl --system
\end{lstlisting}

验证参数生效（返回值应为 1）：
\begin{lstlisting}
sysctl net.bridge.bridge-nf-call-iptables net.bridge.bridge-nf-call-ip6tables net.ipv4.ip_forward
\end{lstlisting}

\textbf{步骤2：安装容器运行时（containerd，所有节点）}

关键：Ubuntu 26.04 默认使用 cgroup v2，必须将 containerd 的 SystemdCgroup 设为 true。

\begin{lstlisting}
apt update
apt install -y containerd
\end{lstlisting}

生成默认配置：
\begin{lstlisting}
mkdir -p /etc/containerd
containerd config default | tee /etc/containerd/config.toml
\end{lstlisting}

设置 cgroup 驱动为 systemd（关键！）：
\begin{lstlisting}
sed -i 's/SystemdCgroup = false/SystemdCgroup = true/' /etc/containerd/config.toml
\end{lstlisting}

配置 pause 沙盒镜像为国内源（关键！）：
\begin{lstlisting}
sed -i 's|sandbox_image =.*|sandbox_image = "registry.aliyuncs.com/google_containers/pause:3.9"|' /etc/containerd/config.toml
\end{lstlisting}

添加 pinned\_images（containerd 1.7+ 双保险）：
\begin{lstlisting}
cat <<EOF | sudo tee -a /etc/containerd/config.toml
[plugins.'io.containerd.cri.v1.images'.pinned_images]
  Sandbox = 'registry.aliyuncs.com/google_containers/pause:3.9'
EOF
\end{lstlisting}

重启并启用 containerd：
\begin{lstlisting}
systemctl restart containerd
systemctl enable containerd
\end{lstlisting}

验证配置：
\begin{lstlisting}
containerd config dump | grep -i sandbox
# 应输出 registry.aliyuncs.com/google_containers/pause:3.9
\end{lstlisting}

\textbf{步骤3：安装 kubeadm、kubelet、kubectl（所有节点）}

\begin{lstlisting}
apt install -y apt-transport-https ca-certificates curl
\end{lstlisting}

添加 Kubernetes APT 仓库（v1.36）：
\begin{lstlisting}
curl -fsSL https://pkgs.k8s.io/core:/stable:/v1.36/deb/Release.key | sudo gpg --dearmor -o /etc/apt/keyrings/kubernetes-apt-keyring.gpg
echo 'deb [signed-by=/etc/apt/keyrings/kubernetes-apt-keyring.gpg] https://pkgs.k8s.io/core:/stable:/v1.36/deb/ /' | sudo tee /etc/apt/sources.list.d/kubernetes.list
\end{lstlisting}

安装组件：
\begin{lstlisting}
apt update
apt install -y kubelet kubeadm kubectl
apt-mark hold kubelet kubeadm kubectl   # 防止意外升级
\end{lstlisting}

\textbf{步骤4：修复 kubelet systemd 路径（所有节点）}

通过 apt 安装的 kubelet 位于 /usr/bin/kubelet，但 systemd drop-in 文件可能默认指向 /usr/local/bin/kubelet，导致启动失败（status=203/EXEC）。

确认 kubelet 位置：
\begin{lstlisting}
which kubelet   # 应输出 /usr/bin/kubelet
\end{lstlisting}

修改 drop-in 文件：
\begin{lstlisting}
sed -i 's|/usr/local/bin/kubelet|/usr/bin/kubelet|g' /etc/systemd/system/kubelet.service.d/10-kubeadm.conf
\end{lstlisting}

重载并启动：
\begin{lstlisting}
systemctl daemon-reload
systemctl start kubelet
systemctl enable kubelet
\end{lstlisting}

此时 kubelet 会因为缺少 /var/lib/kubelet/config.yaml 而报错，这是正常的，该文件由 kubeadm init 生成。

\textbf{步骤5：初始化集群（控制平面节点）}

\textbf{5.1 预拉取镜像（可选）}
\begin{lstlisting}
kubeadm config images pull --image-repository=registry.aliyuncs.com/google_containers
\end{lstlisting}

\textbf{5.2 执行初始化}
\begin{lstlisting}
kubeadm reset -f   # 清除之前残留（首次可跳过）
kubeadm init \
    --image-repository=registry.aliyuncs.com/google_containers \
    --pod-network-cidr=10.244.0.0/16 \
    --v=5
\end{lstlisting}

保存输出的 kubeadm join 命令（用于工作节点）。

\textbf{5.3 配置 kubectl}
\begin{lstlisting}
mkdir -p $HOME/.kube
cp -i /etc/kubernetes/admin.conf $HOME/.kube/config
chown $(id -u):$(id -g) $HOME/.kube/config
\end{lstlisting}

\textbf{步骤6：安装网络插件（Flannel）}

\begin{lstlisting}
wget https://github.com/flannel-io/flannel/releases/latest/download/kube-flannel.yml
sed -i 's|docker.io/flannel|registry.aliyuncs.com/google_containers/flannel|g' kube-flannel.yml
kubectl apply -f kube-flannel.yml
\end{lstlisting}

等待所有 Flannel Pod 变为 Running：
\begin{lstlisting}
kubectl get pods -n kube-flannel -o wide -w
\end{lstlisting}

\textbf{步骤7：加入工作节点}

在每个工作节点上完成 第 1~4 步，然后执行控制平面输出的 kubeadm join 命令。

如果 token 过期，在控制平面重新生成：
\begin{lstlisting}
kubeadm token create --print-join-command
\end{lstlisting}

\textbf{步骤8：验证集群}

\begin{lstlisting}
kubectl get nodes
# 所有节点应为 Ready
kubectl get pods --all-namespaces
# 所有系统 Pod 应为 Running
\end{lstlisting}

\textbf{步骤9：高可用配置：HAProxy + Keepalived}

目标：为 API Server 提供统一的 VIP 入口（192.168.4.210:6443），实现负载均衡和故障自动转移。

\textbf{9.1 在所有控制平面节点安装 HAProxy 和 Keepalived}

\begin{lstlisting}
apt update
apt install -y haproxy keepalived
\end{lstlisting}

\textbf{9.2 配置 HAProxy（所有节点相同）}

编辑 /etc/haproxy/haproxy.cfg：

\begin{lstlisting}
global
    log /dev/log local0
    maxconn 4096
    user haproxy
    group haproxy
    daemon

defaults
    log global
    mode tcp
    option tcplog
    option dontlognull
    retries 3
    timeout connect 5s
    timeout client 30s
    timeout server 30s

frontend kubernetes-api
    bind 192.168.4.210:6443   # 绑定到 VIP
    mode tcp
    option tcplog
    default_backend kubernetes-backend

backend kubernetes-backend
    mode tcp
    option tcp-check
    balance roundrobin
    server k8sprod1 192.168.4.200:6443 check fall 3 rise 2
    server k8sprod2 192.168.4.201:6443 check fall 3 rise 2
    server k8sprod3 192.168.4.202:6443 check fall 3 rise 2
\end{lstlisting}

注意：bind 必须绑定到 VIP 地址（192.168.4.210:6443），而不是 0.0.0.0，否则与 kube-apiserver 端口冲突。

\textbf{9.3 配置 Keepalived}

在 k8sprod1（MASTER，priority 150）：

\begin{lstlisting}
tee /etc/keepalived/keepalived.conf > /dev/null <<EOF
vrrp_instance VI_1 {
    state MASTER
    interface ens18                # 替换为实际网卡名
    virtual_router_id 51
    priority 150
    advert_int 1
    authentication {
        auth_type PASS
        auth_pass k8s_vip_123
    }
    virtual_ipaddress {
        192.168.4.210/24
    }
    notify /etc/keepalived/notify.sh
}
EOF
\end{lstlisting}

在 k8sprod2（BACKUP，priority 120）：

\begin{lstlisting}
tee /etc/keepalived/keepalived.conf > /dev/null <<EOF
vrrp_instance VI_1 {
    state BACKUP
    interface ens18
    virtual_router_id 51
    priority 120
    advert_int 1
    authentication {
        auth_type PASS
        auth_pass k8s_vip_123
    }
    virtual_ipaddress {
        192.168.4.210/24
    }
    notify /etc/keepalived/notify.sh
}
EOF
\end{lstlisting}

在 k8sprod3（BACKUP，priority 110）：

\begin{lstlisting}
tee /etc/keepalived/keepalived.conf > /dev/null <<EOF
vrrp_instance VI_1 {
    state BACKUP
    interface ens18
    virtual_router_id 51
    priority 110
    advert_int 1
    authentication {
        auth_type PASS
        auth_pass k8s_vip_123
    }
    virtual_ipaddress {
        192.168.4.210/24
    }
    notify /etc/keepalived/notify.sh
}
EOF
\end{lstlisting}

\textbf{9.4 创建 notify 脚本（所有节点）}

\begin{lstlisting}
tee /etc/keepalived/notify.sh > /dev/null <<'EOF'
#!/bin/bash
case $3 in
    MASTER)
        systemctl start haproxy
        ;;
    BACKUP|FAULT)
        systemctl stop haproxy
        ;;
esac
exit 0
EOF

chmod +x /etc/keepalived/notify.sh
\end{lstlisting}

\textbf{9.5 修改 kube-apiserver 绑定地址（所有控制平面节点）}

编辑 /etc/kubernetes/manifests/kube-apiserver.yaml，在 command 中添加 --bind-address=<本机物理IP>：

\begin{lstlisting}
    - --bind-address=192.168.4.200   # k8sprod1
    # - --bind-address=192.168.4.201   # k8sprod2
    # - --bind-address=192.168.4.202   # k8sprod3
\end{lstlisting}

保存后，kubelet 会自动重启 API Server。

\textbf{9.6 重新生成 API Server 证书（包含 VIP）}

在 k8sprod1 上执行：

\begin{lstlisting}
rm /etc/kubernetes/pki/apiserver.crt /etc/kubernetes/pki/apiserver.key
kubeadm init phase certs apiserver \
    --apiserver-advertise-address=192.168.4.200 \
    --apiserver-cert-extra-sans="192.168.4.210,192.168.4.201,192.168.4.202"
systemctl restart kubelet
\end{lstlisting}

验证证书：

\begin{lstlisting}
openssl x509 -in /etc/kubernetes/pki/apiserver.crt -text -noout | grep -A 1 "Subject Alternative Name"
# 应包含 IP Address:192.168.4.210
\end{lstlisting}

\textbf{9.7 修改 kubeadm-config ConfigMap}

\begin{lstlisting}
kubectl edit configmap kubeadm-config -n kube-system
\end{lstlisting}

在 ClusterConfiguration 中添加：

\begin{lstlisting}
controlPlaneEndpoint: "192.168.4.210:6443"
\end{lstlisting}

\textbf{9.8 启动服务（所有节点）}

\begin{lstlisting}
systemctl enable keepalived haproxy
systemctl restart keepalived   # HAProxy 由 notify 脚本自动管理
\end{lstlisting}

验证 VIP：

\begin{lstlisting}
ip a s | grep 192.168.4.210   # 只在 MASTER 节点显示
curl -k https://192.168.4.210:6443/healthz    # 应返回 ok
\end{lstlisting}

\textbf{步骤10：提升为多控制平面}

使用 VIP 地址加入其他控制平面节点：

\textbf{10.1 在 k8sprod1 生成新凭证}

\begin{lstlisting}
kubeadm token create --print-join-command
kubeadm init phase upload-certs --upload-certs
\end{lstlisting}

\textbf{10.2 在 k8sprod2/k8sprod3 上清理并加入}

\begin{lstlisting}
kubeadm reset -f
rm -rf /etc/kubernetes /var/lib/etcd

kubeadm join 192.168.4.210:6443 \
    --token <新token> \
    --discovery-token-ca-cert-hash sha256:<新hash> \
    --control-plane \
    --certificate-key <新密钥> \
    --v=5
\end{lstlisting}

\textbf{10.3 修改新控制平面节点的 API Server 绑定地址}

在每个新控制平面节点上，编辑 /etc/kubernetes/manifests/kube-apiserver.yaml，添加 --bind-address=<本机物理IP>。

\textbf{10.4 验证}

\begin{lstlisting}
kubectl get nodes
# 所有节点 ROLES 应为 control-plane，STATUS 应为 Ready
\end{lstlisting}

\textbf{步骤11：集群升级（kubeadm upgrade）}

重要原则：Kubernetes 不支持跨多个次要版本（Minor Version）升级。例如，从 v1.34 升级到 v1.36，必须先经过 v1.35。升级前务必备份 etcd 和 /etc/kubernetes 目录。

\textbf{11.1 升级前准备}

备份集群关键数据：

\begin{lstlisting}
# 备份 etcd
ETCDCTL_API=3 etcdctl snapshot save /root/etcd-snapshot-$(date +%Y%m%d).db \
  --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \

  --key=/etc/kubernetes/pki/etcd/server.key

# 备份 Kubernetes 配置

# 检查集群健康状况

\textbf{11.2 控制平面节点升级}

一次只升级一个控制平面节点，建议从主节点（k8sprod1）开始。

步骤 1：升级 kubeadm

\begin{lstlisting}
# 查看可用的目标版本
apt-cache madison kubeadm

# 替换 <目标版本> 为实际版本号，例如 1.36.3-1.1
apt-mark unhold kubeadm && \
apt-get update && \
apt-get install -y kubeadm='<目标版本>' && \
apt-mark hold kubeadm
\end{lstlisting}

步骤 2：检查升级计划

\begin{lstlisting}
kubeadm upgrade plan
\end{lstlisting}

步骤 3：执行控制平面升级

\begin{lstlisting}
kubeadm upgrade apply v<目标版本>
# 例如：kubeadm upgrade apply v1.36.3
\end{lstlisting}

步骤 4：升级 kubelet 和 kubectl

\begin{lstlisting}
apt-mark unhold kubelet kubectl && \
apt-get update && \
apt-get install -y kubelet='<目标版本>' kubectl='<目标版本>' && \
apt-mark hold kubelet kubectl
\end{lstlisting}

步骤 5：重启 kubelet

\begin{lstlisting}
systemctl daemon-reload
systemctl restart kubelet
\end{lstlisting}

步骤 6：升级其他控制平面节点

对 k8sprod2 和 k8sprod3 重复上述 1、4、5 步骤，但在第 3 步应使用：

\begin{lstlisting}
kubeadm upgrade node
\end{lstlisting}

\textbf{11.3 工作节点升级}

所有控制平面节点升级完成后，再逐个升级工作节点。

步骤 1：腾空节点

在控制平面节点上执行：

\begin{lstlisting}
kubectl drain <工作节点名称> --ignore-daemonsets --delete-emptydir-data
# 例如：kubectl drain k8sprod4
\end{lstlisting}

步骤 2：在工作节点上升级 kubeadm

\begin{lstlisting}
apt-mark unhold kubeadm && \
apt-get update && \
apt-get install -y kubeadm='<目标版本>' && \
apt-mark hold kubeadm
\end{lstlisting}

步骤 3：升级节点配置

\begin{lstlisting}
kubeadm upgrade node
\end{lstlisting}

步骤 4：升级 kubelet 和 kubectl

\begin{lstlisting}
apt-mark unhold kubelet kubectl && \
apt-get update && \
apt-get install -y kubelet='<目标版本>' kubectl='<目标版本>' && \
apt-mark hold kubelet kubectl
\end{lstlisting}

步骤 5：重启 kubelet

\begin{lstlisting}
systemctl daemon-reload
systemctl restart kubelet
\end{lstlisting}

步骤 6：恢复节点调度

在控制平面节点上执行：

\begin{lstlisting}
kubectl uncordon <工作节点名称>
\end{lstlisting}

对每个工作节点重复上述步骤。

\chapter{附录：关键注意事项}

\begin{itemize}
    \item \textbf{cgroup 驱动}：Ubuntu 26.04 仅支持 cgroup v2，containerd 必须设置 SystemdCgroup = true。
    \item \textbf{pause 镜像}：必须单独配置，--image-repository 不影响它。
    \item \textbf{kubelet 路径}：务必检查 systemd 单元文件中的路径。
    \item \textbf{网络插件}：必须在初始化后尽快安装。
    \item \textbf{VIP 绑定}：HAProxy 必须绑定到 VIP 地址，而非 0.0.0.0。
    \item \textbf{证书 SAN}：API Server 证书必须包含 VIP 地址。
    \item \textbf{notify 脚本}：确保在所有控制平面节点上配置，实现 HAProxy 自动启停。
    \item \textbf{防火墙}：放行 6443、6444（VIP 健康检查）、VRRP 协议（IP 协议号 112）。
    \item \textbf{升级策略}：每次升级前备份 etcd；一次只升级一个控制平面节点；工作节点需先 drain 再升级。
\end{itemize}
