## 系统依赖

* Bacula 已在 Linux、FreeBSD、MacOSX、Solaris 以及许多其他 Linux/Unix 系统上编译并运行。 

* 需要 GNU C++ 2.95 或更高版本来编译。可以尝试使用其他编译器和旧版本，但只能自行操作。我们已成功使用 GNU C++ 4.1.3版本编译并使用了 Bacula 。注意，通常 GNU C++ 是一个独立的包（例如 RPM）来自 GNU C ，因此需要同时加载它们。在 Red Hat 系统上，C++ 编译器是 `gcc-c++ rpm` 包的一部分。 

* Bacula 可能需要某些第三方软件包。除了 MySQL 和 PostgreSQL 之外，它们都可以在 `depkgs` 和 `depkgs1` 版本中找到。然而，大多数当前的 Linux 和 FreeBSD 系统都提供这些作为系统软件包。 

* The minimum versions for each of the databases supported by Bacula are:支持的数据库每个最小版本 

* 如果想要一款适用于 Windows 的 File daemon，[建议咨询网站:www.baculasystems.com](http://www.baculasystems.com)，该网站提供的二进制文件费用非常有限。如果您是 Bacula Systems 的客户，Windows 二进制文件包含在您的基本订阅中。 

* Bacula requires a good implementation of pthreads to work.Bacula 需要对线程进行良好的实施才能发挥作用。某些 BSD 系统的情况并非如此。 

* The source code has been written with portability in mind and is mostly POSIX compatible. Thus porting to any POSIX compatible operating system should be relatively easy.源代码在编写过程中考虑到了可移植性,[POSIX](https://www.bacula.org/15.0.x-manuals/en/main/Acronyms.html#bacronyms_posix)且大多与POSIX兼容。因此,[POSIX](https://www.bacula.org/15.0.x-manuals/en/main/Acronyms.html#bacronyms_posix)将移植到任何兼容POSIX的操作系统上应相对容易。 

* GNOME 控制台程序在 GNOME 2.x 下开发并进行测试。GNOME 1.4 不再受支持。 

* The wxWidgets Console program is developed and tested with the latest stable ANSI or Unicode version of wxWidgets (2.6.1). It works fine with the Windows and GTK+-2.x version of wxWidgets, and should also work on other platforms supported by wxWidgets.

  wxWidgets 控制台程序使用最新的稳定 ANSI 或 Unicode 版本([2](http://www.wxwidgets.org).6.1)进行开发和测试。它与 wxWidgets 的 Windows [和 ](https://www.bacula.org/15.0.x-manuals/en/main/Acronyms.html#bacronyms_gtk)GTK +-2.x 版本兼容,也适用于 wxWidgets 支持的其他平台。 

* Tray Monitor 程序是为 GTK+-2.x 开发的。It needs GNOME less or equal to 2.2, KDE greater or equal to 3.1 or any window manager supporting the FreeDesktop system tray standard.需要的 GNOME 小于或等于 2.2、KDE 大于或等于 3.1,或任何支持 [FreeDesktop ](http://www.freedesktop.org/Standards/systemtray-spec)系统托盘标准的窗口管理器。 

* 如果要启用命令行编辑和历史记录，则需要包含 `/usr/include/termcap.h` 以及已加载的 termcap 或 ncurses 库（`libtermcap-devel` 或 `ncurses-devel`）。 

* If you want to use DVD as backup medium, you will need to download the dvd+rw-tools 5.21.4.10.8, apply the patch that is in the patches directory of the main source tree to make these tools compatible with Bacula, then compile and install them. There is also a patch for dvd+rw-tools version 6.1, and we hope that the patch is integrated into a later version.如果要使用 DVD 作为备份介质，则需要下载 `dvd+rw-tools 5.21.4.0.8` ，然后将主源树补丁目录中的补丁应用patches，以使这些工具与 Bacula 兼容,然后进行编译并安装。dvd+rw-toolsdvd+rw-tools 版本 6.1 还有一个补丁,我们希望该补丁能集成到后续版本中。请勿使用发行版提供的 dvd+rw-tool ，除非您确定该工具包含补丁。没有补丁的 dvd+rw-tool 将无法与 Bacula 配合使用。 DVD media is not recommended for serious or important backups because of its low reliability. 由于其可靠性较低，不建议使用 DVD 媒体进行认真或重要的备份。

## 支持的磁带驱动器

Bacula 使用标准操作系统调用（ read，write，ioctl ）来连接到磁带驱动器。As a consequence, it relies on having a correctly written [OS](https://www.bacula.org/15.0.x-manuals/en/main/Acronyms.html#bacronyms_os) tape driver. 因此,[OS](https://www.bacula.org/15.0.x-manuals/en/main/Acronyms.html#bacronyms_os)它依赖于拥有一个正确写入的操作系统磁带驱动程序。众所周知，Bacula 在 FreeBSD、Linux、Solaris 和 Windows 设备上与 SCSI 磁带驱动程序配合得非常好，并且在其他 *nix 设备上可能也能正常工作。

On Linux the OnStream drive, which uses the OSST driver is one such example, and it is known to work with Bacula. In addition a number of such tape drives (i.e. [OS](https://www.bacula.org/15.0.x-manuals/en/main/Acronyms.html#bacronyms_os) drivers) seem to work on Windows systems. However, non-SCSI tape drives (other than the OnStream) that use ide-scsi, ide-tape, or other non-scsi drivers do not function correctly with Bacula (or any other demanding tape application) as of today (April 2007). If you have purchased a non-SCSI tape drive for use with Bacula on Linux, there is a good chance that it will not work. We are working  with the kernel developers to rectify this situation, but it will not be resolved in the near future. 

 Generally any modern tape drive (i.e. after 2010) will work out of the box with Bacula using the standard Bacula Device specification in the bacula-sd.conf file. 

 Even if your drive is on the list below, please check the [Tape Testing section](https://www.bacula.org/15.0.x-manuals/en/problems/Bacula_Frequently_Asked_Que.html#blb:btape1) of the Bacula Enterprise Problems Resolution guide for procedures that you can use to verify if your tape drive will work with Bacula. If your drive is in fixed block mode, it may appear to work with Bacula until you attempt to do a restore and Bacula wants to position the tape. You can be sure only by following the procedures suggested above and testing. 

 It is very difficult to supply a list of supported tape drives, or drives that are known to work with Bacula because of limited feedback (so if you use Bacula on a different drive, please let us know). ##    Unsupported Tape Drives 



 Previously OnStream IDE-SCSI tape drives did not work with Bacula. As of Bacula version 1.33 and the osst kernel driver version 0.9.14 or later, they  now work. Please see the testing chapter as you must set a fixed block  size. 

 QIC tapes are known to have a number of particularities (fixed block size, and one EOF rather than two to terminate the tape). As a  consequence, you will need to take a lot of care in configuring them to  make them work correctly with Bacula.

##  FreeBSD Users Be Aware!!!



 Unless you have patched the pthreads library on FreeBSD 4.11 systems, you will lose data when Bacula spans tapes. This is because the unpatched pthreads library fails to return a warning status to Bacula that the end of the tape is near. This problem is fixed in FreeBSD systems released after 4.11. Please see the [Tape testing section](https://www.bacula.org/15.0.x-manuals/en/problems/Autochanger_Resource.html#blb:FreeBSDTapes) of Bacula Enterprise Problems Resolution guide for important information on how to configure your tape drive for compatibility with Bacula.

##  Tape Specifications

 If you want to know what tape drive to buy that will work with Bacula, we really cannot tell you. However, we can say that if you are going to buy a drive, you should try to avoid DDS drives. The technology is  rather old and DDS tape drives need frequent cleaning. DLT drives are  generally much better (newer technology) and do not need frequent  cleaning. 

 Below, you will find a table of [DLT](https://www.bacula.org/15.0.x-manuals/en/main/Acronyms.html#bacronyms_dlt) and [LTO](https://www.bacula.org/15.0.x-manuals/en/main/Acronyms.html#bacronyms_lto) tape specifications that will give you some idea of the capacity and  speed of modern tapes. The capacities that are listed are the native  tape capacity without compression. All modern drives have hardware  compression, and manufacturers often list compressed capacity using a  compression ration of 2:1. The actual compression ratio will depend  mostly on the data you have to backup, but I find that 1.5:1 is a much  more reasonable number (i.e. multiply the value shown in the table by  1.5 to get a rough average of what you will probably see). The transfer  rates are rounded to the nearest GB/h. All values are provided by  various manufacturers. 

 The Media Type is what is designated by the manufacturers and you are not required to use (but you may) the same name in your Bacula conf resources.

 如果你想知道该购买哪种胶带驱动器,可以使用 巴库拉我们真的不能告诉你。但是,我们可以说,如果你打算购买驱动器,应尽量避免使用DDS驱动器。这项技术相当陈旧,而DDS胶带驱动装置需要频繁清洁。DLT驱动器通常性能更好(技术较新),无需频繁清洁。 

 下面,[DLT](https://www.bacula.org/15.0.x-manuals/en/main/Acronyms.html#bacronyms_dlt)[LTO](https://www.bacula.org/15.0.x-manuals/en/main/Acronyms.html#bacronyms_lto)您将找到一张DLT和LTO胶带规格的表格,可让您了解现代胶带的容量和速度。列出的容量是无需压缩的原生磁带容量。所有现代驱动器均采用硬件压缩,制造商通常使用2:1的压缩率来列出压缩容量。实际压缩比主要取决于你需要备份的数据,但我发现1.5:1是一个更合理的数值(即将表中显示的数值乘以1.5,以大致计算出你可能看到的数据)。转账费率为最接近的GB/h。所有价值均由各制造商提供。
