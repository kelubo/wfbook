# 分布式数据库原理

## 概述

分布式数据库是将数据存储在多个物理节点上的数据库系统，通过网络协作提供数据服务。

---

## 一、分布式数据库架构

### 1.1 架构类型

```
分布式数据库架构
        │
        ├─ 分片式架构
        │   ├─ 水平分片：按行划分数据
        │   ├─ 垂直分片：按列划分数据
        │   └─ 混合分片：结合水平和垂直分片
        │
        ├─ 复制式架构
        │   ├─ 主从复制：一个主节点，多个从节点
        │   ├─ 多主复制：多个主节点，相互复制
        │   └─ 链式复制：节点按链状复制
        │
        └─ 混合架构：分片+复制
```

### 1.2 CAP定理

```
CAP定理
        │
        ├─ C（一致性）：所有节点同时看到相同的数据
        ├─ A（可用性）：每个请求都能得到响应
        └─ P（分区容错性）：网络分区时系统仍能运行

        CAP权衡
        │
        ├─ CP：放弃可用性，保证一致性和分区容错性（如MongoDB）
        ├─ AP：放弃一致性，保证可用性和分区容错性（如Cassandra）
        └─ CA：放弃分区容错性，保证一致性和可用性（如传统数据库）
```

### 1.3 BASE理论

```
BASE理论（对CAP的扩展）
        │
        ├─ BA（基本可用）：系统在部分故障时仍能提供基本服务
        ├─ S（软状态）：允许数据存在中间状态
        └─ E（最终一致性）：数据最终会达到一致状态
```

## 二、数据分片策略

### 2.1 分片方式

```
数据分片方式
        │
        ├─ 哈希分片
        │   ├─ 根据主键哈希值分配到不同节点
        │   ├─ 优点：数据分布均匀
        │   └─ 缺点：扩容困难
        │
        ├─ 范围分片
        │   ├─ 根据主键范围分配到不同节点
        │   ├─ 优点：范围查询效率高
        │   └─ 缺点：数据可能分布不均
        │
        ├─ 列表分片
        │   ├─ 根据预定义列表分配数据
        │   ├─ 优点：便于管理
        │   └─ 缺点：需要预先规划
        │
        └─ 复合分片：结合多种分片方式
```

### 2.2 分片键选择

```
分片键选择原则
        │
        ├─ 高基数：分片键应具有大量唯一值
        ├─ 均匀分布：数据应均匀分布到各个节点
        ├─ 查询模式：分片键应与常见查询模式匹配
        └─ 稳定性：分片键不应频繁变化
```

### 2.3 分片管理

```
分片管理
        │
        ├─ 分片发现：发现和管理分片节点
        ├─ 分片迁移：数据在分片间迁移
        ├─ 分片扩容：添加新分片节点
        └─ 分片缩容：移除分片节点
```

## 三、分布式事务

### 3.1 事务模型

```
分布式事务模型
        │
        ├─ 两阶段提交（2PC）
        │   ├─ 阶段1：准备阶段，协调者询问参与者是否就绪
        │   ├─ 阶段2：提交阶段，协调者通知参与者提交或回滚
        │   ├─ 优点：保证强一致性
        │   └─ 缺点：性能差，协调者单点故障
        │
        ├─ 三阶段提交（3PC）
        │   ├─ 在2PC基础上增加了预提交阶段
        │   └─ 减少了阻塞问题
        │
        └─ 最终一致性事务
            ├─ 使用消息队列实现最终一致性
            ├─ 优点：性能高，可用性好
            └─ 缺点：数据可能短暂不一致
```

### 3.2 分布式锁

```
分布式锁实现方式
        │
        ├─ 数据库锁：利用数据库唯一约束
        ├─ Redis锁：利用Redis原子操作
        ├─ ZooKeeper锁：利用ZooKeeper节点特性
        └─ etcd锁：利用etcd的CAS操作
```

## 四、一致性协议

### 4.1 Paxos协议

```
Paxos协议
        │
        ├─ 角色：提案者、接受者、学习者
        ├─ 阶段：准备阶段、接受阶段
        ├─ 特点：保证一致性，容错性强
        └─ 应用：ZooKeeper、etcd
```

### 4.2 Raft协议

```
Raft协议
        │
        ├─ 角色：领导者、跟随者、候选者
        ├─ 阶段：选举阶段、日志复制阶段
        ├─ 特点：比Paxos简单，易于实现
        └─ 应用：Consul、TiDB
```

### 4.3 Gossip协议

```
Gossip协议
        │
        ├─ 节点随机与其他节点交换信息
        ├─ 最终达到一致性
        ├─ 特点：去中心化，容错性强
        └─ 应用：Cassandra、Redis Cluster
```

## 六、分布式数据库设计考虑

```
分布式数据库设计考虑
        │
        ├─ 数据一致性：选择强一致性还是最终一致性
        ├─ 可用性：系统故障时的服务保证
        ├─ 性能：读写吞吐量和延迟
        ├─ 可扩展性：水平扩展能力
        ├─ 容错性：故障恢复能力
        └─ 成本：硬件和运维成本
```

---

## 总结

分布式数据库是应对大规模数据存储和高并发访问的关键技术，通过数据分片、复制和一致性协议实现高可用、高性能的数据服务。
